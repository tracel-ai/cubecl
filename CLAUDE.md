# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

CubeCL is a multi-platform compute language extension for Rust: a `#[cube]` Rust function is JIT-compiled on demand to CUDA, HIP, Metal, SPIR-V, WGSL, or CPU SIMD. This repo holds the language, the proc-macro frontend, the IR, the optimizer, and the per-platform runtimes. Kernel *libraries* (matmul, reductions, convolution, attention, etc.) live in a separate repo, [cubek](https://github.com/tracel-ai/cubek), and [Burn](https://burn.dev) is the downstream DL framework that drove the design.

This is a `no_std`-first workspace (`edition = "2024"`, MSRV 1.95). Most crates declare `#![no_std]` and gate `std` behind a feature.

## Commands

All dev workflows go through `cargo xtask` (a `.cargo/config.toml` alias). Prefer it over raw cargo for anything touching CI parity.

```sh
cargo xtask validate                         # run before every PR — full check + test suite
cargo xtask check --target all lint          # clippy + fmt, auto-FIXES issues (commit first)
cargo xtask ci --target crates lint          # same checks but report-only (no auto-fix), reproduces CI
cargo xtask test --target crates all         # unit + integration + doc tests over all crates
cargo xtask test --target crates unit        # unit tests only (also: integration, documentation)
cargo xtask test --target crates all --ci    # excludes cubecl-cuda / cubecl-hip (no GPU on CI)
cargo xtask test --target all all            # crates + examples
```

`--target` is one of `crates`, `examples`, or `all`. `check` auto-fixes and prompts before modifying code; `ci` only reports. CI itself runs `xtask check audit`, `xtask check format`, `xtask check lint`, typos, and the test matrix.

### Running a single test

The runtime test suites are generated by macros (see "Testing model" below), so the tests live under a backend crate. Target them with normal cargo filters and pick a backend via features:

```sh
# Run one generated test on the wgpu backend
cargo test -p cubecl-wgpu tests::slice -- --nocapture

# Verbose tracing for one test (see PERFORMANCE.md)
RUST_LOG=trace cargo test -p cubecl-cuda --features tracing,test-log/default \
  tests::identity::f16_ty::test_large -- --nocapture
```

### Running examples

Each `examples/*` crate selects a backend by Cargo feature, not by code:

```sh
cargo run --example gelu --features cuda      # or wgpu, cpu
```

Note: the root `Cargo.toml` sets `debug = 0` for faster compiles. To debug, set `CARGO_PROFILE_DEV_DEBUG=true` (see CONTRIBUTING.md for the VS Code/LLDB setup).

## Architecture

### The compilation pipeline (the central idea)

`#[cube]` does **not** lower Rust to IR directly. It rewrites the annotated function into a *new* Rust function that, when called, **builds the IR at runtime** (kernel expansion). Comptime is therefore just ordinary Rust executing at expansion time, and `comptime!` results are baked into the IR as constants. Generics, traits, helper fns, and modules all work because the kernel is still Rust until the IR-building call fires.

```
#[cube] fn  ──▶  expanded Rust fn  ──▶  CubeCL IR  ──▶  {WGSL, CUDA, SPIR-V, HIP, Metal, MLIR}
```

Kernels are JIT-compiled: only the variants you actually launch are generated, and results are cached on-device (and shippable with the binary as a warm autotune/compilation cache).

### Crate layers (dependency direction, bottom → top)

- **cubecl-common** — shared primitives, `future`, no-std utilities. Depended on by nearly everything.
- **cubecl-ir** — the intermediate representation (scopes, instructions, types, vectorization). The contract between frontend and all compilers.
- **cubecl-zspace** — shape/stride (`Shape`, `Strides`) math.
- **cubecl-macros** / **cubecl-macros-internal** — the `#[cube]` proc macro and friends (`src/{parse,generate,expression,statement,scope}`). This is where Rust → expanded-Rust rewriting happens.
- **cubecl-runtime** — backend-agnostic runtime: `ComputeClient`/`ComputeServer`, memory management, autotune, compilation cache, storage, streams. Not bound to the `#[cube]` frontend — external kernels (e.g. PTX) can plug in here.
- **cubecl-core** — the frontend types and launch machinery built on ir + runtime + macros. `frontend/` (cube types, `cmma`, `comptime`, `plane`, topology, branches), `codegen/` (compiler/integrator), `compute/` (launcher, builder). Re-exports much of the public surface via `prelude`. Hosts the shared `runtime_tests/`.
- **cubecl-opt** — IR optimization passes, consumed by the compiler backends.
- **Compiler backends**: **cubecl-cpp** (CUDA/HIP C++ codegen, shared by cuda+hip), **cubecl-spirv** (SPIR-V).
- **Runtime backends** (top layer, what users enable as features): **cubecl-cuda**, **cubecl-hip**, **cubecl-wgpu** (WGSL/SPIR-V/Metal-MSL via wgpu), **cubecl-cpu** (MLIR via `tracel-llvm`).
- **cubecl-std** — the in-repo standard library of reusable cube code (tensor helpers, etc.).
- **cubecl** — the umbrella crate users depend on; feature flags (`cuda`, `wgpu`/`vulkan`/`webgpu`/`metal`, `hip`/`rocm`, `cpu`) pull in the matching backend.

A backend crate generally depends on: cubecl-core + cubecl-runtime + cubecl-common + its compiler crate (cpp or spirv) + cubecl-std.

### The four-axis hardware model

Kernels are portable because hardware is abstracted as four orthogonal axes of parallelism, each a kernel-level constant (capitalized, Rust const syntax): **Vector** (SIMD lanes), **Plane** (warp/subgroup/SIMD-group, lockstep), **CubeDim** (block/workgroup/threadgroup, shared mem + sync), **CubeCount** (grid). Good kernels read the runtime's maxima at comptime and adapt — **never hardcode** `PLANE_DIM == 32` (breaks on AMD's 64, meaningless on CPU where it's 1). Vectorization/CubeDim/CubeCount are chosen at launch; plane size is set by the runtime. See the topology table in `README.md` for the CUDA/WebGPU/Metal name mapping.

### Testing model

Backend correctness tests are written **once** in `crates/cubecl-core/src/runtime_tests/` (and `cubecl-std`) and exposed behind the `export_tests` feature via `testgen!` / `testgen_all!` macros. Each backend crate instantiates them in its `src/lib.rs` `#[cfg(test)] mod tests` block, parameterized over the dtypes that backend supports — e.g. cubecl-wgpu has separate `tests`, `tests_spirv`, and `tests_msl` modules with different type lists. To add a runtime test, add it under `runtime_tests/` and wire it into `mod.rs`; it then runs on every backend automatically.

## Conventions

- `expect()` messages describe the *successful* expectation, not the error ("The X should be set by Y"). See CONTRIBUTING.md.
- Always gate `tracing::instrument` and any tracing code behind the `tracing` feature (see PERFORMANCE.md).
- New code must keep `no_std` compatibility unless the crate is std-only — guard std usage behind the `std` feature.
- `typos` runs in CI; intentional/technical identifiers go in `typos.toml`.
